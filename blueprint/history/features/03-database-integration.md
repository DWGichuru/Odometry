# Feature: Database integration

**From build-plan:** feature 3
**Status:** not started

## Goal

Install Prisma ORM, define the complete PostgreSQL schema on Neon, create the initial migration, and seed the database with a test user and mock shifts so feature 5 can read real data on day one. No app code consumes the database yet -- this is pure infrastructure.

## Design reference

Not a visual feature. No prototype applies.

## In scope

- Install `prisma` and `@prisma/client` (dev dependency + runtime dependency)
- Create `prisma/schema.prisma` with the full data model from `project-overview.md`: `User`, `Credential`, `Subscription`, `Shift`, `Account`, `Session`, `VerificationToken`
- Prisma enums for `Platform` and `EntrySource` matching the locked TypeScript types
- `.env` and `.env.example` with `DATABASE_URL` (Neon PostgreSQL connection string)
- `lib/prisma.ts` -- shared Prisma client singleton (dev hot-reload safe)
- Initial Prisma migration (`prisma migrate dev`)
- `prisma/seed.ts` -- creates one test user + mock shifts; runnable via `tsx prisma/seed.ts` and `"prisma": { "seed": "tsx prisma/seed.ts" }` added to `package.json`
- Add `postinstall` script: `prisma generate` (so deploys auto-generate the client)
- Update `AGENTS.md` commands if needed (Prisma commands: `prisma migrate dev`, `prisma generate`)

## Out of scope

- App code that reads/writes shifts from the database (feature 5)
- Authentication wiring (feature 4 -- the NextAuth adapter tables exist but NextAuth is not installed or configured)
- Subscription/Stripe logic (feature 8)
- Production deployment with `prisma migrate deploy` (feature 10)
- Any route, page, or UI that shows database data
- Replacing mock data in the dashboard or shift form -- they still use `lib/mock-data.ts` until feature 5

## Build loop

Build one step at a time, never the whole feature at once.

1. Plan mode lays out the step before any code.
2. The AI implements just that step.
3. It shows the diff (not full files); you read it and understand it.
4. You approve, then choose whether to commit a checkpoint or roll straight on.
   Checkpoints are optional; `/complete` makes the real feature-level commit at the end.

Never accept a step you haven't read. If a diff is too big to review, the step was too big, so split it.

## Data / contracts

The Prisma schema **is the contract** for features 4, 5, 6, 7, and 8. Every model, column, type, relation, and cascade rule must match the project overview data model exactly.

Key constraints:
- `User.id` is a cuid string (the Prisma default). `Shift.userId` is a string FK with `onDelete: Cascade`.
- `Credential.userId` and `Subscription.userId` are unique (one-to-one with User).
- `Subscription.status` is an enum: `trialing`, `active`, `past_due`, `canceled`.
- `Shift.platform` is a Prisma enum matching the locked `Platform` TS enum.
- `Shift.entrySource` is a Prisma enum matching the locked `EntrySource` TS enum.
- NextAuth adapter tables (`Account`, `Session`, `VerificationToken`) follow the NextAuth v5 Prisma adapter convention for column names and compound keys.

## Notes for the AI

- The user must provide a Neon PostgreSQL `DATABASE_URL` before step 2. Format: `postgresql://user:password@host/dbname?sslmode=require`. Add it to `.env` and a placeholder to `.env.example`.
- `prisma generate` creates the client in `node_modules/.prisma/client/` -- add this directory to `.gitignore`.
- The `lib/prisma.ts` singleton pattern: `const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }; export const prisma = globalForPrisma.prisma ?? new PrismaClient(); if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;`. This prevents connection exhaustion in dev from HMR.
- The seed script should be idempotent -- check if data already exists before inserting. Or use `deleteMany` + `createMany` for a clean reseed. For this feature, a simple upsert approach: if the test user exists, skip; otherwise create everything.
- The seed uses `@prisma/client` directly, so `tsx` is needed to run TypeScript in Node. `tsx` should already be available (it's a transitive dep commonly used with Prisma seed). If not, install it as a dev dependency.
- Do not wire any app code to Prisma yet. The dashboard, shift form, and mock data stay untouched.
- Do not install NextAuth or configure it -- the adapter tables exist purely to make the migration ready for feature 4.
